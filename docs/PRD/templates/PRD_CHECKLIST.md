# PRD 품질 검증 체크리스트 (Quality Validation Checklist)

> PRD 작성 완료 후, 아래 체크리스트를 사용하여 품질을 검증하세요.
> 필수 항목을 모두 충족해야 리뷰에 제출할 수 있습니다.

---

## 1. 문서 구조 & 메타 정보

- [ ] 문서 버전, 작성일, 작성자, 상태가 기재되어 있는가
- [ ] 변경 이력(Changelog)이 관리되고 있는가
- [ ] 마크다운 형식으로 작성되었는가 (.md 확장자)

---

## 2. 배경 & 문제 정의

- [ ] 현재 상태가 데이터/메트릭 기반으로 기술되어 있는가
- [ ] **왜 지금** 이 문제를 해결해야 하는지 근거가 있는가
- [ ] 솔루션이 아닌 **문제** 중심으로 기술되어 있는가
- [ ] 현재 상태의 한계가 구체적으로 기술되어 있는가

---

## 3. 목표 & 비목표

- [ ] 모든 목표에 **정량적 성공 기준**이 포함되어 있는가
- [ ] 성공 기준에 **측정 방법**이 명시되어 있는가
- [ ] **비목표(Non-Goals)** 가 명시적으로 정의되어 있는가
- [ ] 비목표마다 "왜 하지 않는지" 근거가 있는가

---

## 4. 범위 & 가정

- [ ] 포함 범위와 제외 범위가 명확히 구분되어 있는가
- [ ] 전제 조건(Assumptions)에 **검증 방법과 시점**이 명시되어 있는가
- [ ] 기술적/비즈니스/운영 제약 조건이 기술되어 있는가

---

## 5. 사용자 & 시나리오

- [ ] 행위 주체(사용자, 관리자, 시스템)가 정의되어 있는가
- [ ] 핵심 시나리오에 사전/사후 조건이 포함되어 있는가
- [ ] **예외 흐름**이 정의되어 있는가 (해피 패스만이 아님)

---

## 6. 기능 요구사항

- [ ] 각 요구사항에 **고유 ID**가 부여되어 있는가 (REQ-001 등)
- [ ] **MoSCoW 우선순위**가 지정되어 있는가 (Must/Should/Could/Won't)
- [ ] 각 요구사항에 **수락 기준(Acceptance Criteria)** 이 포함되어 있는가
- [ ] 수락 기준이 **검증 가능한** 형태로 작성되어 있는가
- [ ] **입력(Inputs)/출력(Outputs)/동작(Behavior)** 이 명시되어 있는가
- [ ] **엣지 케이스**가 식별되고 처리 방법이 기술되어 있는가
- [ ] 의존성이 ID 기반으로 명시되어 있는가
- [ ] 기능(WHAT)과 구조(HOW)가 **분리**되어 기술되어 있는가

---

## 7. 비기능 요구사항

- [ ] 성능 요구사항에 **정량적 목표값**이 있는가 (예: P95 ≤ 200ms)
- [ ] 신뢰성 목표가 정의되어 있는가 (가용성, MTTR 등)
- [ ] 확장성 전략이 기술되어 있는가
- [ ] 보안 요구사항이 정의되어 있는가 (인증, 인가, 암호화, API 보안)
- [ ] **민감 데이터 처리 방법**이 정의되어 있는가
- [ ] 관측성(로그, 메트릭, 트레이스, 알람)이 정의되어 있는가

---

## 8. 의존성 그래프

- [ ] 모든 모듈 간 의존성이 **명시적으로** 선언되어 있는가
- [ ] **암묵적 의존성**이 없는가
- [ ] **순환 의존성**이 없는가
- [ ] Phase별로 그룹핑되어 위상 정렬이 가능한가
- [ ] 각 모듈의 Public Interface가 명확한가

---

## 9. API & 데이터

- [ ] API 엔드포인트별 인증/접근 제어가 명시되어 있는가
- [ ] 에러 모델이 정의되어 있는가
- [ ] 핵심 데이터 모델/엔터티가 정의되어 있는가
- [ ] 데이터 흐름이 시각화되어 있는가

---

## 10. 리스크 & 이슈

- [ ] 리스크마다 **발생 확률, 영향도, 완화 전략**이 있는가
- [ ] 오픈 이슈에 **결정 시한과 의사결정자**가 지정되어 있는가
- [ ] 기술 결정에 **대안과 선택 근거**가 기술되어 있는가

---

## 11. 구현 로드맵

- [ ] Phase별 **진입/종료 조건**이 정의되어 있는가
- [ ] Phase별 **산출물**이 명확한가
- [ ] 마일스톤이 시각화되어 있는가
- [ ] 예상 기간이 현실적인가

---

## 12. 롤아웃 & 마이그레이션

- [ ] 단계적 배포 전략이 정의되어 있는가
- [ ] Feature Flag 전략이 있는가 (필요한 경우)
- [ ] 역호환성 보장 방안이 있는가
- [ ] 데이터 마이그레이션 계획과 **롤백 계획**이 있는가

---

## 13. 테스트 & 검증

- [ ] 테스트 피라미드 비율이 정의되어 있는가
- [ ] 커버리지 요구사항이 명시되어 있는가
- [ ] 모듈별 핵심 테스트 시나리오가 구체적으로 기술되어 있는가
- [ ] 성능/보안 테스트 계획이 포함되어 있는가

---

## 14. 이해관계자 & 승인

- [ ] 이해관계자 승인란이 포함되어 있는가 (PM, Tech Lead, Design 등)
- [ ] 각 승인자의 역할과 승인 상태가 명시되어 있는가

---

## 15. 요구사항 추적성

- [ ] 각 요구사항에 고유 ID가 있는가
- [ ] 수락 기준이 체크리스트(`- [ ]`) 형태로 되어 있는가
- [ ] 의존성이 ID 기반으로 연결되어 있는가
- [ ] Phase별 의존성 그래프가 포함되어 있는가
- [ ] 요구사항이 이슈 트래커 티켓과 연결 가능한 형태인가
- [ ] 요구사항이 **적절한 크기의 작업 단위**로 작성되어 있는가
